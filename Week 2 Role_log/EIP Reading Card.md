# MIP 结构化阅读卡片：MIP-8 Page-ified Storage State
## 提案基础信息
1. 提案编号：**MIP-8**
2. 提案全称：Page-ified Storage State（分页存储状态）
3. 原文链接：https://forum.monad.xyz/c/mips/8
4. 所属生态：Monad 改进提案（MIP），针对 Monad L1 底层存储性能优化

## 一、背景问题（提案出台前现存痛点）
1. **EVM 存储单元与磁盘硬件读写单元严重错配**
磁盘硬件最小读写单位是4KB物理页，一页可容纳128个EVM标准32字节存储槽。但传统EVM Merkle Trie仅以单个存储槽为最小操作粒度，合约里逻辑连续的数组、结构体变量会被哈希打散，分散在磁盘不同页面中。
2. **随机IO过多，大幅拖累节点运行效率**
当合约批量读取连续存储数据时，节点需要反复加载多张磁盘页，产生大量低效随机IO，直接拉长区块执行、全节点同步耗时。
3. 原有Gas计费无法反映真实硬件成本
Gas仅按单次读写指令收费，不区分“首次加载全新磁盘页”和“读取内存中已缓存页面”的硬件开销差距，导致存储密集型合约手续费定价失真。
4. 状态树提交、快照同步效率低下
修改任意单个存储槽都要更新一整条Merkle路径，不支持同一页面内批量变更合并，节点归档、历史状态快照同步速度慢、带宽占用高。

## 二、核心方案（简单拆解）
1. 将磁盘4KB物理页升级为EVM一等存储单元，每128个连续存储槽组成一页，让逻辑连续的合约变量自然落在同一张磁盘页，贴合硬件读写逻辑。
2. 重构Merkle状态树结构：以完整页面作为树的叶子节点，同一页面内批量修改仅需更新一条Merkle路径，减少海量哈希计算。
3. 分层差异化Gas计费，贴合硬件真实开销：
   - 首次访问空白磁盘页：收取一次性高额Page Load Gas，对应磁盘读取成本；
   - 同一页面内后续读写：仅收取极低基础Gas，仅消耗内存资源，无磁盘IO。
4. 支持页级批量变更与快照优化，区块结束后按页面批量提交修改；同步节点可按页面批量拉取状态，减少同步流量与等待时间。
5. 完全上层兼容：Solidity合约、ABI、读写指令无需任何修改，仅底层数据库与状态树逻辑升级，开发者无迁移成本。

## 三、影响对象
1. **全节点/归档节点运维者**
利好：磁盘随机IO显著减少，区块执行、链同步速度提升，低配硬件也可稳定运行节点，硬件运维成本下降。
2. **合约开发者（DeFi、NFT、数组密集型项目）**
利好：连续数组、结构体读写的Gas成本大幅降低，循环高频存储类交易手续费更便宜，无需改动业务代码即可受益。
3. 普通MON持币/交易用户
利好：存储密集型交易整体Gas消耗下降，日常链上操作平均手续费降低。
4. 区块生产者、验证节点
利好：区块打包、状态根计算耗时缩短，链整体交易吞吐上限提升。
5. Monad底层客户端开发团队
利空：需要完整重写状态数据库、Merkle Trie逻辑，存储引擎改造有较大开发、测试工作量。
6. 链浏览器、链上数据索引器
利好：历史快照、状态查询支持分页批量读取，数据索引速度明显提升。

## 四、关键术语整理
1. Storage Slot：EVM最小存储单元，固定32字节，用于存放合约变量
2. Disk Page：磁盘4KB物理读写页，硬件最小IO单元，单页容纳128个Storage Slot
3. Page-ified Storage：分页存储，以硬件物理页作为状态树基础操作单元
4. Merkle Trie：传统以太坊系链原生状态树，MIP-8将叶子粒度从单槽改为整页
5. Page Load Gas：首次加载全新磁盘页面时收取的一次性Gas费用，匹配磁盘IO成本
6. QMDB：Monad自研Merkle数据库，为本提案配套底层存储实现
7. State Sync：全节点同步整条链全部历史账户与合约状态的流程

## 五、争议与潜在风险
1. Gas模型公平性争议
反对观点：仅零散写入离散存储槽的稀疏合约，会频繁触发高额Page Load Gas，单次交易手续费大幅上涨。提案回应：稀疏存储本身硬件IO开销更高，Gas规则如实还原硬件成本，符合资源消耗匹配计费的公平原则。
2. 主网升级迁移复杂度风险
链上全部历史状态都是单槽粒度Merkle结构，升级需要完整状态转换快照；若采用增量迁移，会同时存在两套状态树格式，极易引发同步故障。
3. 第三方生态工具兼容隐患
离线分析工具、本地测试节点、区块浏览器若不跟进分页存储逻辑，读取历史状态会出现数据错乱、校验失败。
4. 上线初期共识分叉风险
全新页粒度Merkle树逻辑复杂度高，上线前需要海量测试网验证；若存在Merkle根计算漏洞，会引发全网共识分叉。

## 六、产品&工程启发
1. 底层存储抽象层设计必须对齐物理硬件特性，若上层逻辑与底层IO模型割裂，会产生持续性隐性性能损耗。
2. 链上计费、Gas模型应锚定真实硬件资源开销，而非单纯按指令计数，实现手续费与资源消耗匹配。
3. 底层批量优化做到上层完全透明，合约层无需改动，能最大程度降低整个生态的迁移阻力。
4. 公链扩容不只有并行交易、分片方案，存储层IO优化是低成本、高收益的性能提升路径。
5. 去中心化发展的核心瓶颈之一是节点同步成本，分页、批量快照机制可以降低归档节点运维门槛，有利于网络去中心化。
